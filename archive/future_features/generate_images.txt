  - Quiet hours / hide logic so it doesn't drive you insane. Easy, ~30 lines.
  - Persistence of mood/last-seen across restarts. Easy, ~20 lines.

  Total code: 500–600 LOC. Not scary, just more surface area. I could write it in a few hours.

  Art (the real cost)

  To look good you need, minimum:
  - Habitat background (room with desk, terminal, coffee, plant, window, bookshelf) — 1 image
  - Bob idle — 4 frames (breathing/blinking loop)
  - Bob walking — 4 frames (left), 4 frames (right)
  - Bob sitting at desk — 2 frames
  - Bob typing at terminal — 4 frames
  - Bob drinking coffee — 3 frames
  - Bob stretching — 4 frames
  - Bob sleeping — 2 frames (breathing loop)
  - Bob excited/happy — 2 frames
  - Bob confused/error — 2 frames
  - Bob greeting wave — 4 frames

  That's ~35 sprites in a consistent style. Pixel art sprite sheets are the single worst thing to generate with Midjourney/DALL-E — you get 35 sprites of 35 different Bobs. Consistency is
  the killer. Real paths for this are:

  1. Hand-draw or commission them — Aseprite + someone who knows pixel art. Best quality, real time investment (or money).
  2. Find a free asset pack on itch.io that roughly fits, restyle the character. Fast but Bob won't be Bob.
  3. AI-generate + heavy manual touchup in Aseprite. Painful.

  The art problem is why I didn't recommend this one. It's not that the code is hard — it's that without proper assets, it looks sad, and with proper assets, you're spending a weekend in
  Aseprite.

  Design iteration (the sneaky hard part)

  A pet that just sits there is dead. A pet that moves constantly is annoying. Finding the beat where Bob feels alive without being distracting takes real tuning. You'll watch him for 20
  minutes, realize he stretches too often, change the timer, realize he never walks, change the pathing, realize the coffee break only fires once a day and nobody sees it… this is
  design-test-iterate work, not just code.

      - Same flat schema we use for everything else: { prompt: string, output_path?: string, reference_images?: [string] }
      - Internally calls gemini-2.5-flash-image exactly like the script that just made your mockup
      - Saves PNG to ~/Pictures/Bob/ (or wherever) and returns the path
      - Bob can then read_file to "see" what he made or open it for you
      - Approval policy: probably none (read-only-ish, no destructive side effects beyond writing to a designated folder), or modal if you want to gate it
      - Cost: ~80 lines of Swift. A GeminiImageProvider protocol mirroring SearchProvider, an ImageGenerationTool, registry hookup. One afternoon.
      - Killer feature: because nano-banana accepts reference images, Bob could be told "make me a t-shirt design with you (Bob) holding a cup of coffee" and pass his own sprite sheet as
    the reference. The character stays consistent. That's wildly fun.
    2. MCP client (the bigger architectural move)
      - This is on the v2 deferred list in CLAUDE.md and it's a real build
      - Adds a generic plugin surface so ANY MCP server (image gen, GitHub, Linear, filesystem, etc.) shows up as Bob tools
      - Bob's tool registry becomes dynamic: built-ins + N MCP servers' worth of advertised tools
      - Requires: MCP stdio/SSE transport, schema discovery, approval policy that handles unknown tools gracefully (probably defaulting to modal), config UI for adding/removing servers,
    sandboxing of which tools each server is allowed to expose
      - Cost: meaningfully bigger. Probably a long weekend. The hard parts are the approval story (how do you trust a random MCP server's tools?) and the schema translation (MCP advertises
    richer schemas than Ollama's flat-tool format will accept — needs flattening or filtering).

    My recommendation: do #1 first, #2 later.

    Reasoning:
    - Image generation is the single most fun capability to add right now and it's a 1-day job. Ship the dopamine immediately.
    - Doing the native tool first proves out the patterns (provider protocol, tool registration, approval) on something concrete before generalizing.
    - MCP is a real architectural commitment — it changes the trust model of the whole app. Worth doing properly, not rushed.
    - The same GeminiImageProvider you build for the native tool can later be exposed THROUGH an MCP server adapter if you decide to standardize.

    Heads-up on a few things:
    - The Gemma4 tool format is flat. If you want Bob to pass an array of reference images, you'll need to fake it as a comma-separated string, because nested/array params in tool schemas
    crashed Gemma when we tested earlier. Single optional reference image is safer.
    - Persist generated images somewhere durable and tag them with the prompt + timestamp in SQLite, so the Tool Activity window can show thumbnails of what Bob's made.
    - Cost: nano-banana isn't free. Track usage, maybe show running cost in the menu bar. Cheap per call but easy to rack up if Bob gets enthusiastic.
    - For approvals, generation is benign but prompt injection from chat could make Bob generate something you don't want. Probably worth defaulting generate_image to modal until you've
    watched it for a while.

    So: yes to image generation (ship it as a native 5th tool soon), yes to MCP eventually (real v2 work, plan it properly). Don't conflate the two — knock out the high-fun, low-effort one
    first.
